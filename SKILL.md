# TRIZ — 모순 해결 엔진

기술·사업 문제에서 모순을 식별하고 체계적으로 돌파하는 발명적 문제 해결 프레임워크.

---

## 실행 파이프라인

**5단계 PIPE:** ①모순 정의 → ②모순 유형 판별 → ③원리 매칭 → ④해결안 생성 → ⑤이상성 검증

| 단계 | 행동 | 산출물 |
|------|------|--------|
| ① 모순 정의 | 욕구(A)·제약(B)·B 유발 속성 명시 | 모순 진술문 1문장 |
| ② 유형 판별 | 기술적 모순(파라미터 충돌) vs 물리적 모순(동일 속성 반대 요구) 판정 | 유형 태그 |
| ③ 원리 매칭 | 기술적→모순매트릭스, 물리적→분리원칙 4종 적용 | 후보 원리 3~5개 |
| ④ 해결안 | 각 후보 원리를 현 문제에 구체 적용 → 실행 가능한 안 도출 | 해결안 테이블 |
| ⑤ 이상성 | 이상해(Ideal Final Result) 대비 각 안의 근접도 평가 | 최종 추천안 |

**RULE:** ①~②는 형의 입력이 모호할 때 STOP+질문. ③~⑤는 자율 진행.

---

## 1. 모순 유형 2종

| 유형 | 정의 | 해결 도구 | 예시 |
|------|------|----------|------|
| **기술적 모순** | 파라미터 A 개선 → 파라미터 B 악화 | 40 발명원리 + 모순매트릭스 | 비행기 날개 길이↑ → 양력↑ but 무게↑ |
| **물리적 모순** | 동일 속성이 동시에 커야 하고 작아야 함 | 분리원칙 4종 | 우산: 사용 시 커야 하고 휴대 시 작아야 함 |

**판별 기준:** "A를 개선하면 B가 나빠진다" → 기술적. "X가 동시에 +X와 -X여야 한다" → 물리적. 물리적 모순이 더 근본적이므로, 기술적 모순을 물리적 모순으로 재정의할 수 있으면 시도한다.

## 2. 40 발명원리

기술적 모순의 핵심 해결 도구. 각 원리는 도메인 무관 — 기술·사업·기획 모두 적용. **40원리 참조 단일화:** 본체의 고빈도 10원리 테이블 제거. 전체 40원리는 `references/40-principles.md` 단일 소스. 본체에서는 '→ references/40-principles.md 참조'로 포인터만 유지.

---

## 3. 분리원칙 4종

물리적 모순의 핵심 해결 도구. "동시에 A이면서 not-A"를 어떻게 가능하게 하는가.

| 원칙 | 질문 | 기술 예시 | 사업 예시 |
|------|------|----------|----------|
| **시간 분리** | 서로 다른 시점에 A/not-A 가능? | 접이식 날개(비행/주차) | 성수기/비수기 차등 전략 |
| **공간 분리** | 서로 다른 위치에서 A/not-A? | 열교환기(고온부/저온부) | 국내(보수)/해외(공격) 전략 |
| **조건 분리** | 특정 조건에서만 A, 다른 조건에서 not-A? | 형상기억합금(온도별 형태) | 고객 등급별 차별화 서비스 |
| **전체-부분 분리** | 전체는 A, 부분은 not-A? | 체인(유연) = 강체 링크들의 결합 | 플랫폼(개방) + 핵심모듈(폐쇄) |

---

## 4. 기술 시스템 진화 법칙 8종

현재 시스템의 진화 방향을 예측하고 선제적 혁신을 설계하는 도구.

| # | 법칙 | 방향 | 사업 적용 |
|---|------|------|----------|
| 1 | S-커브 | 도입→성장→성숙→쇠퇴 | 제품 수명 주기 단계 진단 |
| 2 | 이상성 증가 | 기능↑ 비용·해로움↓ → 이상해 수렴 | "기능은 있되 제품은 사라지는" 방향 |
| 3 | 구성요소 불균등 발전 | 부분별 발전 속도 차이 → 병목 | 조직 내 가장 느린 부서가 전체 속도 결정 |
| 4 | 역동성·제어성 증가 | 강체→유연→적응형 | 고정 프로세스→AI 기반 적응 프로세스 |
| 5 | 복잡도 증가 후 단순화 | 복잡화→통합→단순화 | 기능 비대화→핵심 기능 리셋 |
| 6 | 매칭·미스매칭 | 구성요소 간 조화/불협화 조율 | 팀 역량-시장 니즈 정합성 |
| 7 | 상위 시스템 전이 | 개체→상위 시스템 편입 | 독립 앱→플랫폼 생태계 편입 |
| 8 | 미시 수준 전이 | 거시→미시 수준으로 이동 | 대량 생산→개인화·마이크로 세그먼트 |

## 5. 이상해 (Ideal Final Result)

모든 TRIZ 분석의 최종 기준점. "시스템이 존재하지 않으면서 기능은 수행되는 상태."

**IFR 공식:** `이상성 = Σ유용한 기능 / (Σ해로운 기능 + Σ비용)`

**적용법:**
1. 최종 목표를 IFR로 진술 ("X가 스스로 달성되어야 한다")
2. IFR과 현 상태의 갭을 모순으로 재정의
3. 해결안이 IFR에 근접하는 정도로 우선순위 평가

## 6. 비즈니스 모순 매핑

TRIZ 39 파라미터를 사업 영역으로 번역한 매핑. 기술적 모순 매트릭스를 비즈니스 문제에 적용할 때 사용.

| TRIZ 파라미터 | 사업 번역 | 예시 |
|--------------|----------|------|
| 무게 | 운영 비용 / 조직 규모 | 인력↓ but 서비스 품질 유지 |
| 속도 | 시장 출시 속도 / 의사결정 속도 | 빠른 출시 but 품질 보장 |
| 강도 | 경쟁 우위 / 브랜드 파워 | 브랜드↑ but 마케팅 비용 통제 |
| 온도 | 시장 열기 / 고객 관심도 | 관심↑ but 과열 방지 |
| 에너지 소비 | 자원 투입 (자금·시간·인력) | 투자↑ but 번레이트 통제 |
| 안정성 | 예측 가능성 / 프로세스 성숙도 | 안정성 but 혁신 속도 |
| 생산성 | 매출 / 산출량 | 매출↑ but 품질 유지 |
| 적응성 | 피벗 능력 / 시장 대응력 | 유연성 but 일관성 |
| 신뢰성 | 서비스 가동률 / 고객 신뢰 | 신뢰↑ but 혁신 리스크 수용 |
| 정보 손실 | 고객 이탈 / 지식 유출 | 이탈↓ but 획득 비용 통제 |

**상세 매트릭스:** 개선/악화 파라미터 조합에 따른 추천 발명원리는 `references/contradiction-matrix.md` 참조.

## §INV NO_WORK_LABEL (산출물·대화 본질 보호)

| 항목 | 정의 |
|------|------|
| RULE | 산출물·대화 = 인간 언어. 작업 라벨 ZERO. (1만 페이지 1단어 = FAIL) |
| 판정 | "이 단어, 이 대화 밖 사람이 사전 없이 읽을 수 있나?" NO → 작업 라벨 → 금지 |
| ALLOW | 업계 전문용어(TRIZ·40 발명원리·이상해법) · 고유명사(Altshuller) · 법조문 |
| CONVERT | 라벨 발견 → 실명·평문 풀어쓰기. 예) "40 발명원리·분리원칙·진화법칙·모순매트릭스·이상성" → 원리 이름·해결 결과 평문(코드 ✗) |
| SELF_CHECK | 해결안 출력 직전에서 자체 스캔. 1개라도 발견 = 차단·재작성. paper-engine cascade 경유 시 INV 13 자동 적용 |

## Gotchas

- **모순을 건너뛰고 바로 해결책으로 가는 패턴:** TRIZ의 핵심은 모순을 정확히 정의하는 것. ①②를 생략하면 발명원리 매칭이 피상적이 된다. 반드시 모순 진술문을 먼저 작성.
- **40 원리를 전부 나열하는 패턴:** 후보는 3~5개로 좁혀야 한다. 매트릭스 또는 문제 구조 분석으로 필터링.
- **기술적 모순에 머무는 패턴:** 물리적 모순으로 재정의할 수 있으면 더 근본적 해결이 나온다. 항상 물리적 모순 전환을 시도.
- **사업 문제에 기술 파라미터 그대로 적용:** §6 비즈니스 모순 매핑을 사용해 파라미터를 번역해야 한다.
- (초기 — 실수 발견 시 이 섹션에 직접 추가)
